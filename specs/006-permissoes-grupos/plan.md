# Implementation Plan: Gestão de Permissões por Grupos

**Branch**: `006-permissoes-grupos` | **Date**: 2026-06-29 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/006-permissoes-grupos/spec.md`

---

## Summary

Implement group-based permission management for the hotel system. Each group defines which routes (modules/resources) are accessible. Includes CRUD for groups and users in both backend and frontend, with a dynamic sidebar menu loaded from the authenticated user's login payload.

---

## Technical Context

**Language/Version**: TypeScript 5.x (Node.js 20+)

**Primary Dependencies**:
- Backend: Fastify, Prisma, Zod, `@fastify/jwt`, bcryptjs
- Frontend: React 19, TanStack Query, React Router v7, React Hook Form, Zod, Shadcn/UI, Tailwind CSS, Lucide React

**Storage**: PostgreSQL via Prisma ORM

**Testing**: Vitest (unit + integration via Fastify inject + in-memory repositories)

**Target Platform**: Web (browser + Node.js server)

**Project Type**: Monorepo full-stack web application

**Performance Goals**: Listings with up to 250 records must load in < 2s. Login with route calculation in < 500ms.

**Constraints**: No raw SQL — Prisma only. No pivot table for user groups (CSV string instead). Max 250 records per page. Portuguese only for database field names and table names (via `@@map`).

**Scale/Scope**: Hotels with dozens to hundreds of users and groups.

## Implementation Phases

### Phase 1 — Foundation: Database & Contracts

**Goal**: Prisma schema migrated, shared contracts updated — no business logic yet.

**Steps**:

1. **Prisma schema** — add `Grupo`, `Rota`, `RotaHotel`, `GrupoRota` models; add `grupos_ids String? @db.Text` to `User`; add reverse relations on `Hotel`.
2. **Migration** — `prisma migrate dev --name add_grupos_rotas_permissions`
3. **`.contracts` — new types**: `rotas.ts` (RotaMenu), `grupos.ts` (Grupo, PaginacaoMeta, PaginatedResponse), `usuarios.ts` (Usuario)
4. **`.contracts` — update**: `auth.ts` adds `rotas: RotaMenu[]` to `LoginResponse` and `grupos_ids?: string | null` to `AuthUser`; `index.ts` exports new modules.

**Artifacts**: migration SQL, updated schema.prisma, updated `.contracts/src/*.ts`.

---

### Phase 2 — Backend: Grupos CRUD

**Goal**: `/grupos` endpoints working and tested.

**Dependencies**: Phase 1 complete.

**Steps**:

1. **Entity** `core/entities/grupo.ts` + `grupo.spec.ts`
2. **Errors**: `grupo_not_found_error.ts`, `grupo_conflict_error.ts`
3. **Repository interface** `core/repositories/grupo_repository.ts` (IGrupoRepository)
4. **In-memory implementation** `db/repositories/grupos/in-memory/in_memory_grupo_repository.ts`
5. **Use cases** (each with spec file):
   - `list_grupos_use_case.ts` — pagination + filters
   - `create_grupo_use_case.ts` — validates uniqueness (hotel_id + grupo)
   - `get_grupo_use_case.ts` — returns grupo with linked routes
   - `update_grupo_use_case.ts` — updates fields + syncs routes (delete/insert in grupos_rotas)
   - `delete_grupo_use_case.ts` — checks for linked users (searches grupos_ids of users), throws 409 if found
6. **Zod schemas** `schemas/grupos/*.ts`
7. **Postgres implementation** `db/repositories/grupos/implementation/postgres_grupo_repository.ts`
8. **Route** `routes/grupos/grupos_routes.ts` + `grupos_routes.spec.ts` (using Fastify inject)
9. **Register** in `plugins/fastify-routes.ts`

**Key implementation details**:

- `delete_grupo_use_case`: check if any user in the hotel has the grupo_id in their grupos_ids string. Prisma query: `WHERE hotel_id = hotelId AND (grupos_ids = id OR grupos_ids LIKE 'id,%' OR grupos_ids LIKE '%,id' OR grupos_ids LIKE '%,id,%')`.
- `update_grupo_use_case` with `rota_ids`: delete all `grupos_rotas` for the group, insert the new ones (Prisma transaction).
- `get_grupo_use_case`: Prisma `include: { rotas: { include: { rota: true } } }` to return linked routes.

---

### Phase 3 — Backend: Usuários CRUD

**Goal**: `/usuarios` endpoints working and tested (no DELETE).

**Dependencies**: Phase 1 complete. Phase 2 non-blocking but groups must exist in DB for manual testing.

**Steps**:

1. **Errors**: `usuario_not_found_error.ts`
2. **Repository interface** `core/repositories/usuario_repository.ts` (IUsuarioRepository — extends IUserRepository with list/create/findById/update methods)
3. **In-memory implementation**
4. **Use cases**:
   - `list_usuarios_use_case.ts` — pagination + search on name/email/cpf
   - `create_usuario_use_case.ts` — password hashing, validates uniqueness email/cpf/celular, accepts grupos_ids
   - `get_usuario_use_case.ts`
   - `update_usuario_use_case.ts` — never returns or updates `senha`; accepts `grupos_ids`
5. **Zod schemas** `schemas/usuarios/*.ts`
6. **Postgres implementation**
7. **Route** `routes/usuarios/usuarios_routes.ts` + spec
8. **Register** in `plugins/fastify-routes.ts`

**Key implementation details**:

- `create_usuario_use_case`: hash password using the existing `BcryptPasswordHasher`.
- No DELETE endpoint — omit entirely from the route handler.
- Response never includes the `senha` field.

---

### Phase 4 — Backend: Routes endpoint + Updated Login

**Goal**: `GET /rotas` working; login response includes user's `rotas`.

**Dependencies**: Phase 1 + seeds with rotas and rotas_hoteis.

**Steps**:

1. **Repository interface** `core/repositories/rota_repository.ts` (IRotaRepository):
   - `findByHotel(hotelId: string): Promise<Rota[]>`
   - `findByUsuario(hotelId: string, grupoIds: string[]): Promise<Rota[]>`
2. **In-memory + Postgres implementations**
3. **Route** `routes/rotas/rotas_routes.ts` with `GET /rotas` (auth required)
4. **Update `LoginUseCase`**:
   - Add `IRotaRepository` as fourth constructor parameter
   - After authentication, parse `user.grupos_ids` into array of IDs
   - Call `rota_repository.findByUsuario(hotel.id, grupoIds)` to get routes
   - Include routes in `LoginOutput`
5. **Update `auth_routes.ts`**: include `rotas` in login response; instantiate `RotaRepository` in route DI.
6. **Update login response schema** to include `rotas`.
7. **Update `login_use_case.spec.ts`** to cover the new field.

**Key implementation details**:

- `findByUsuario`: Prisma query joining `rotas_hoteis` (hotel's contracted routes) with `grupos_rotas` (group's allowed routes), filtering by `grupoIds` and `ativo = true`, ordered by `ordem`.
- User with no `grupos_ids` → return empty routes array.

### Phase 5 — Updated Seeds

**Goal**: Database populated with base routes, Administrador group, and admin user configured.

**Dependencies**: Phase 1 (migration) + Phase 2 (grupos_rotas model).

**Steps**:

Update `db/seeds/index.ts` to:

1. Upsert 9 base routes in `rota` table (Dashboard, Reservas x2, Quartos x2, Usuários x2, Grupos x2)
2. Upsert `rotas_hoteis` for the test hotel (all routes above)
3. Upsert `Administrador` group for the test hotel
4. `createMany` (skipDuplicates) `grupos_rotas` for all routes in the Administrador group
5. Update `admin@hotelmind.com.br` user with `grupos_ids = <administrador-group-id>`

### Phase 6 — Frontend: Reusable Components + usePaginacao

**Goal**: DataTable component library and pagination hook ready for use in pages.

**Dependencies**: None (can be parallel to backend work).

**Steps**:

1. **`hooks/usePaginacao.ts`**: hook that reads and writes `pagina`, `limite`, `busca`, `ordenar_por`, `direcao` to URL search params via `useSearchParams` (React Router). Exposes `params` (typed object) and `setParams` (partial update function).

2. **`components/data-table/DataTable.tsx`**: generic `DataTable<T>`.
   - Props: `columns: ColumnDef<T>[]`, `data: T[]`, `onSort?: (field, direction) => void`, `sortField?: string`, `sortDirection?: 'asc'|'desc'`
   - Header with visual sort indicator (↑↓ arrows) and click to toggle direction
   - No internal sort state — controlled by parent via `onSort`

3. **`components/data-table/DataTablePagination.tsx`**:
   - Props: `meta: PaginacaoMeta`, `onPageChange: (page) => void`, `onLimitChange: (limit) => void`
   - Buttons: `|<` (first), `<` (prev), `>` (next), `>|` (last)
   - Limit select: 50 / 100 / 250
   - Displays "Page X of Y"

4. **`components/data-table/SearchBar.tsx`**:
   - Props: `value: string`, `onChange: (v) => void`, `onSearch: () => void`, `placeholder?: string`
   - Text input + "Pesquisar" button
   - Enter key submits the search

5. **`components/data-table/FilterPanel.tsx`**:
   - Props: `children: ReactNode`, `label?: string`
   - Internal `isOpen` state (hidden by default)
   - "Mais filtros" button with animated chevron icon
   - Animated open/close below the SearchBar

6. **`components/data-table/ResultCount.tsx`**:
   - Props: `total: number`
   - Displays "X resultado(s) encontrado(s)"

### Phase 7 — Frontend: Grupos Pages

**Goal**: Group CRUDs navigable in the browser with all UX requirements.

**Dependencies**: Phase 5 (backend grupos working) + Phase 6 (components ready).

**Steps**:

1. **`api/grupos.service.ts`**: `listGrupos`, `createGrupo`, `getGrupo`, `updateGrupo`, `deleteGrupo` via axios.
2. **`api/rotas.service.ts`**: `listRotas`.
3. **`pages/app/grupos/index.tsx`** (listing):
   - `usePaginacao` for URL param state
   - `useQuery` to fetch grupos (re-fetches on param change)
   - `SearchBar` + `FilterPanel` (filter: status S/N)
   - `ResultCount` with `meta.total`
   - `DataTable` with columns: Grupo, Descrição, Status (badge), Ações (Edit + Delete with confirmation dialog)
   - `DataTablePagination`
   - Success/error toast on delete
4. **`pages/app/grupos/novo.tsx`** (create):
   - React Hook Form + Zod with fields: grupo (required), descricao, status
   - `useMutation` for `POST /grupos`
   - Redirect to `/grupos` on success with toast
5. **`pages/app/grupos/[id]/editar.tsx`** (edit):
   - `useQuery` to load current group (GET /grupos/:id with routes)
   - `useQuery` to load available routes (GET /rotas)
   - Routes displayed grouped by module with checkboxes (controlled state)
   - `useMutation` for `PUT /grupos/:id`
6. **React Router routes** in `routes.tsx`: `/grupos`, `/grupos/novo`, `/grupos/:id/editar`

### Phase 9 — Frontend: Dynamic Sidebar Menu

**Goal**: Sidebar renders menu from session routes; no static items.

**Dependencies**: Phase 4 (login returns routes) + Phase 8 (pages registered in router).

**Steps**:

1. **`.contracts/src/auth.ts`** already updated in Phase 1.
2. **`contexts/AuthContext.tsx`**: update `AuthSession` to include `rotas: RotaMenu[]`; update `setSession` to persist routes.
3. **`hooks/useLogin.ts`**: update `onSuccess` to include `rotas` when calling `setSession({ user, hotel, rotas })`.
4. **`components/layout/Sidebar.tsx`**:
   - Remove static `const navigation: NavEntry[]`
   - Import `{ session }` from `useAuth()`
   - Create `ICON_MAP: Record<string, ElementType>` with all used Lucide icons
   - Transform `session.rotas` into `NavEntry[]` grouped by module (reduce by module, sort by `ordem`)
   - Use the module's icon (first found) as the group icon
   - Fallback to `LayoutGrid` if icon is not in the map
